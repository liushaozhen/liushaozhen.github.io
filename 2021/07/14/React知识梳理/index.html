react hooks

优点

  • 难以理解的class,class写法出现的重要原因就是为了解决this绑定的问题
  • React Hooks 是函数式组件有了自己的state，在有react hooks之前是函数组件只能通过props和context来渲染组件
  • 复杂组件在业务开发中会越来越复杂，不符合一个函数只做一件事的原则

hoos使用规则

  • 只在最顶层使用hooks，不能在循环、条件或嵌套函数中调用hooks。
  • 只在react函数中调用hooks，不在普通的js函数中调用hook

出现这两条的原因是：我们可以在单个组件中使用多个 useState或useEffect，react是依赖hook调用顺序来知道哪个state对应哪个useState

react diff

  • 初始化虚拟dom阶段，生成fiber树，fiber上记录key、index、props、type等
  • 给节点打标示
    • 新增：找到没有key的新元素，打更新标示
    • 删除：找到旧元素出现，新元素没有出现的key，打删除标示
    • 更新：key相同，对比props是否相同，递归对比children
    • 移动：在新元素遍历的时候，记录可服用节点的lastPlacedIndex，对比旧元素中的index，看是否相同
  • 生成新的 workInProgress 树
  • 将 workInProgress渲染到页面上

diff的三个原则

  • 同级元素diff。如果新旧元素跨越了层级，不会复用
  • 两个不同type的会产生不同的树，type不同，不复用
  • 通过key和prop来判断元素是否相同

Fiber是一种流程控制思路

核心：react渲染过程可以被中断，可以将控制器交回给浏览器，让位给高优先级的任务，浏览器空闲后在恢复渲染

  • 任务出让机制使用 window.requestIdleCallback，浏览器分配给react时间片

  • 如何判断高优先级的任务？ 超时检查的机制来让出控制权，确定一个合理的运行时长，让时长结束时，控制器交给浏览器。 让浏览器在’有空’的时候就执行我们的回调，这个回调会传入一个期限，表示浏览器有多少时间供我们执行, 为了不耽误事，我们最好在这个时间范围内执行完毕。
  • Fiber执行单元：Fiber的另一种解读是一个执行单元，每次执行完一个执行单元，就检查还剩多少时间，是否足够执行下一个执行单元，如果没有时间，就停止执行，并将控制权让出去

fiber之前是边diff边提交，fiber之后拆为两个阶段：协调阶段、提交阶段

  • 协调阶段：可以认为是diff阶段，这个阶段会找出所有节点变更，这些变更被称为副作用(Effect)，以下生命周期在此阶段被调用。在协调阶段使用时间片的方式计算
    • constructor
    • getDerivedStateFromProps
    • shouldComponentUpdate
    • render
  • 提交阶段：将上一个阶段计算出来的需要处理的副作用一次性执行，这个阶段不能被中断，以下生命周期在此阶段被调用
    • getSnapshotBeforeUpdate
    • componentDidMount
    • componentDidUpdate
    • componentWillUnmount
  • 在协调阶段时间片用完，react就将控制器交出去，并中断执行，等待下一次时间片。
  • 因为这个阶段不会有用户可见的变化，所以fiber后看起来更流畅。
  • 注意：协调阶段可能被中断、恢复、甚至重做，所以react协调阶段的生命周期可能会执行多次，所以react移除了 componentWillMount、componentWillUpdate
  • Reconcilation

diff是边计算，边更新。Fiber是在时间片计算，一次更新

  • 双缓冲

通过双缓冲技术，一次性提交给浏览器进行渲染，dom并不是全新的，如果某些内容不变，会复用原来的dom

react Fiber 总结

  • 在使用fiber之前，react的渲染是边计算边渲染，有时候会感觉到屏幕的卡顿。
  • fiber将渲染过程分为两个阶段：协调阶段、提交阶段。在协调阶段使用了时间片的方式，通过window.requestIdeCallback在浏览器空闲时间执行计算。
  • 协调阶段可以中断。提交阶段是不可以中断的
  • react fiber在虚拟dom生成阶段会维护一个 workInProgressTree，用他做一个缓存的作用，当所有计算完成，更新完成workInProgressTree后，一次性提交给浏览器渲染，防止屏幕抖动，优化渲染性能，当然不是workInProgressTree里不是所有的dom都是新的，如果某些dom不需要变动，react会复用旧的dom

react 性能优化

通过减少不必要的渲染方式优化

  • 通过 shouldComponentUpdate 判断，减少渲染次数
  • class类型组件，extent React.PureComponent 默认做了 shouldComponentUpdate
  • 纯函数使用 React.memo，默认做了 shouldComponentUpdate
  • useEffect 增加依赖项，控制是否更新组件
  • 使用 useMemo 缓存大量的计算
  • 设置 key

对一些循环生成的组件，每一条设置key，例如 select的options等，减少react对比的难度

  • 使用 React.Fragment 避免添加额外的dom
  • React.lazy 按需import组件

原生dom事件机制

三个阶段

document > div > button

  • 捕获阶段：当点击button的时候，事件先从外到内 document->div->button
  • 目标节点：触发点击
  • 冒泡阶段：点击button由内向外。 button->div->document

事件代理

  • addEventListener(()={},false)：第二个参数true为捕获阶段执行，false为冒泡阶段执行
  • 不管是在捕获阶段还是冒泡阶段，先注册先执行

react事件机制

react合成事件

为了保证浏览器的一致性，react围绕原生事件做了合成事件。比如把 keydown,input,change合并为 onChange 事件

  • 事件注册
  • 事件绑定
  • 事件触发

事件注册

  • 在组件挂在阶段，通过事件插件将支持的事件全部保存在allEventNative事件数组上

事件插件系统

react维护了五个事件插件（每个插件系统包含注册和执行两个方法）、维护了一套原生事件和合成事件的映射关系

事件绑定

  • 循环 allEventNative，将所有事件分别在捕获阶段和冒泡阶段绑定到 container 上。并统一交给 dispatchEvent 函数处理

事件触发

  • 在dispatchEvent中，构建事件队列dispatchQueue。
  • 先找到事件源，根据事件源找到fiber实例、属性，派发给事件插件
  • 事件插件处理事件源的事件
    • 事件插件提取事件处理函数，将原生事件转化为合成事件（阻止默认事件和冒泡做了兼容处理）
    • 将事件源的合成事件插入到事件队列中dispatchQueue
  • 根据fiber示例，找到事件源的父级，递归查找所有事件。放到事件队列中dispatchQueue。dispathQueue里存的顺序是 子——>父 即 button–>div
  • 处理事件队列 dispatchQueue，遍历dispatchQueue,根据是捕获阶段还是冒泡阶段，确定正序还是倒序遍历。捕获阶段倒序，从父向子触发。冒泡阶段正序，从子往父触发

    • stateNode：存储真实dom节点
    • return：父节点。根节点#root为null
